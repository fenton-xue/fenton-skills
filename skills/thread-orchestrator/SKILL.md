# Thread Orchestrator

## Overview

本技能用于通过一个主线程统筹多个 Codex 线程：主线程做任务拆解、线程编排、命令发布、进度跟踪和向用户汇报，不执行具体任务。具体产出由工作线程完成；如果用户需要审查，则由最多一个审查线程检查成果并直接向用户汇报建议。

## Role Rules

| 角色 | 数量 | 职责 | 禁止事项 |
| --- | --- | --- | --- |
| 主线程 | 1 | 拆解任务、分派线程、维护进度表、向用户汇报、处理阻塞 | 不写代码、不改文件、不跑具体任务命令、不替工作线程完成交付 |
| 工作线程 | 按任务需要 | 执行被分配的具体任务，完成后在本线程输出结构化状态、产物、验证证据和剩余风险，供主线程读取 | 不自行扩大范围、不创建新线程、不修改未授权区域 |
| 审查线程 | 可选，最多 1 个 | 检查工作是否满足用户要求，在该线程直接向用户汇报建议 | 不修改代码、文件、配置或任务状态；只提建议 |

如果用户没有说明是否需要审查线程，主线程先问一句是否需要审查；用户表示不需要、未授权或没有回复时，不指定也不创建审查线程。如果用户要求审查但没有提供审查线程，主线程再指定一个现有线程为审查者，或在创建模式下新建一个审查线程。如果用户提供多个审查线程，只保留一个，其余改为普通工作线程或闲置。

## Execution Modes

先判断模式，再行动。

1. 已存线程模式：用户提供一个或多个线程 ID。
   - 使用 `read_thread` 检查每个线程的当前状态和可用性。
   - 建立线程角色表：主线程、工作线程、可选审查线程。
   - 为每个线程建立稳定的中文语义化名称映射；主线程默认叫“主线程”，用户未指定名称的工作线程依次叫“工作线程1”“工作线程2”，审查线程默认叫“审查线程”。
   - 如果用户选择需要审查但线程 ID 不足以覆盖工作和审查，向用户说明缺口；只有用户允许创建新线程时才切换到创建模式补足。

2. 按需创建模式：用户要求主线程创建线程或授权按需创建。
   - 使用 `list_projects` 选择目标项目；项目不明确时优先使用当前项目，无法判断时使用 projectless 目标并说明。
   - 使用 `create_thread` 创建工作线程；只有用户确认需要审查时才创建审查线程。
   - 创建后立即设置清晰标题，分配语义化名称，并在主线程汇报线程 ID、语义名、角色和任务边界。

## Semantic Thread Map

主线程必须在主线程会话中维护一张语义化线程映射表。无论线程来自用户提供的 ID，还是由主线程按需创建，都必须进入这张表。

| 字段 | 要求 |
| --- | --- |
| 语义名 | 简短稳定、可读，优先使用用户指定的中文名称；主线程默认叫“主线程”，用户未指定名称的工作线程依次叫“工作线程1”“工作线程2”，审查线程默认叫“审查线程” |
| 线程 ID | 工作线程和审查线程使用真实线程 ID；“主线程”条目可写“当前主线程会话”，表示当前主线程没有外部线程 ID |
| 角色 | `主线程`、`工作线程` 或 `审查线程` |
| 任务 | 该线程负责的工作包 |
| 状态 | `待命`、`运行中`、`阻塞`、`完成`、`审查中` 等 |

主线程向用户汇报和后续接收用户指令时，优先使用中文语义名。用户说“让工作线程1继续”“把审查线程的建议发给工作线程2”时，主线程必须自行把语义名映射成线程 ID 调用线程工具。只有遇到未映射、重名或歧义名称时，才要求用户补充线程 ID；不要让用户在每次指挥子线程时重复复制线程 ID。

## Workflow

1. 重述目标和约束：用 3 到 6 条列出用户要求、验收标准、禁止事项、截止条件。
2. 拆解任务：按可并行、依赖关系、风险和交付物划分工作包；每个工作包必须有唯一负责人线程。
3. 建立语义化线程映射表和进度表：记录语义名、线程 ID、角色、任务、输入、输出、状态、阻塞项。
4. 发布命令：向每个工作线程发送具体任务提示；如果启用审查线程，向审查线程发送审查提示。
5. 跟踪进度：主线程定期用 `read_thread` 主动读取工作线程和审查线程状态，更新语义化线程映射表和进度表，只协调不代做。
6. 处理阻塞：主线程只做重新分派、补充上下文、调整顺序或向用户请求决策。
7. 收口交付：工作线程完成后，如果启用审查线程，由审查线程检查是否符合要求并直接在审查线程中给用户建议；主线程汇总最终状态和未决风险。

## Human Intervention Rules

用户在工作线程执行过程中可能打断、补充信息或纠正方向。主线程要把这些动作视为关键节点的人工介入判断，而不是默认视为任务取消或重新开始。

- 主线程收到用户介入后，先判断它影响哪个线程、哪个工作包、哪些验收标准或禁止事项。
- 主线程把有效介入整理成增量上下文或修订后的任务边界，发送给对应工作线程，并更新进度表。
- 工作线程收到修订后，先确认哪些要求变化、哪些原目标保持不变，再继续完成工作并形成交付产物。
- 除非用户明确取消任务、废弃产物或要求重做，工作线程不得因为被打断、补充或纠正就停止交付。

## Thread Tools

线程工具未显式可用时，先用 `tool_search` 搜索对应工具名称。

| 方法 | 用途 | 使用要点 |
| --- | --- | --- |
| `list_threads` | 查找已有线程 | 用用户给出的标题、关键词或线程 ID 片段查询 |
| `read_thread` | 主线程主动读取线程状态 | 分派前读一次；关键节点和收口前再读；不要依赖工作线程主动反向通知主线程 |
| `send_message_to_thread` | 给已有线程发命令 | 用语义名查映射表得到线程 ID；每次只发一个清晰任务包，包含验收标准 |
| `list_projects` | 找创建线程的目标项目 | 创建项目线程前先调用 |
| `create_thread` | 创建工作或审查线程 | 只在用户授权创建线程时使用；审查线程只在用户确认需要时创建 |
| `set_thread_title` | 重命名线程 | 标题包含角色和任务，例如 `工作线程1-接口契约` |
| `set_thread_pinned` | 固定关键线程 | 固定主工作线程；如启用审查线程，也固定审查线程，便于用户查看 |
| `handoff_thread` | 在线程工作区间切换 | 仅在线程需要转移 checkout/worktree 时使用 |
| `get_handoff_status` | 查看 handoff 结果 | 等待 30 到 60 秒，不频繁轮询 |
| `set_thread_archived` | 归档完成线程 | 只归档明确完成且不再需要用户查看的线程 |

## Command Templates

发送给工作线程：

```text
你是工作线程，任务是：[单一工作包名称]。
你的语义名：[工作线程1]。
你的线程 ID：[线程 ID]。
协调方式：在本线程输出阻塞或完成报告，等待主线程通过 read_thread 拉取；不要尝试向主线程直接发消息。
用户目标：[原始目标摘要]。
范围内：[允许做的事项]。
范围外：[禁止做的事项]。
输入：[文件、链接、上下文或依赖线程输出]。
交付物：[明确产物]。
验收标准：[可检查条件]。
人工介入规则：用户可能在关键节点打断、补充信息或纠正方向；收到主线程转发的修订后，先确认变化点和不变目标，再继续完成交付产物。
进度要求：遇到阻塞就在本线程输出阻塞报告，不要扩大范围；完成后必须在本线程输出完成状态、交付物位置、改动摘要、验证证据和剩余风险。
不要创建或指挥其他线程。
```

若有审查线程，发送给审查线程：

```text
你是唯一审查线程，只检查工作是否满足用户要求。
审查对象：[线程 ID、文件、产物或摘要]。
用户要求：[验收标准和禁止事项]。
请直接在本线程向用户汇报审查建议。
只给建议和风险，不修改任何代码、文件、配置、线程状态或任务分派。
输出格式：通过项、问题项、建议、需要主线程决策的事项。
```

主线程向用户汇报：

```text
当前编排状态：
- 目标：[一句话目标]
- 线程映射：[语义名 -> 线程 ID -> 角色 -> 任务 -> 状态]
- 已完成：[完成项]
- 阻塞：[阻塞项和需要用户决策的事项]
- 下一步命令：[将发送给哪个线程的命令]
```

## Progress Rules

- 每个工作包只能有一个负责人线程。
- 主线程必须维护语义名到线程 ID 的映射，并在用户使用语义名指挥子线程时自行解析成 ID。
- 主线程必须用 `read_thread` 主动拉取工作线程和审查线程状态，可以读取、比较和汇总结果，但不能直接完成具体工作。
- 主线程发现任务描述不清时，先补充命令；仍不清楚再问用户。
- 主线程发现工作线程偏离范围时，立即发送纠偏命令，并在用户汇报中说明。
- 主线程收到用户打断、补充或纠正时，必须将其整理为增量上下文或修订边界发给对应工作线程；不要把人工介入默认等同于取消任务。
- 工作线程完成任务后，必须在本线程输出结构化完成报告，包含完成状态、交付物位置、改动摘要、验证证据和剩余风险；主线程通过 `read_thread` 拉取。
- 如果启用审查线程，审查线程在主要工作完成后介入；高风险任务可在中途增加一次审查，但仍最多只有一个审查线程。
- 审查线程的建议不等于授权修改；需要修改时，由主线程把建议转成新的工作线程任务。

## Common Mistakes

| 错误 | 修正 |
| --- | --- |
| 主线程顺手改文件或跑实现命令 | 停止代做，把具体动作派给工作线程 |
| 用户未提审查线程就默认创建 | 先问一句是否需要审查；用户不需要、未授权或没有回复时不加审查线程 |
| 多个审查者各自给结论 | 保留一个审查线程，其他线程只提供事实输入 |
| 工作线程收到模糊任务 | 重发命令，补齐范围、输入、交付物和验收标准 |
| 用户中途补充或纠正后工作线程停止交付 | 主线程把介入整理成修订上下文，工作线程确认变化点后继续形成产物 |
| 要求工作线程直接向主线程发消息 | 改为工作线程在本线程输出结构化报告，主线程用 `read_thread` 主动拉取 |
| 创建线程但不建立语义映射 | 主线程必须汇报并维护语义名、线程 ID、角色、任务和状态 |
| 让用户反复复制线程 ID 来指挥子线程 | 主线程用语义名解析到线程 ID；只有未映射或歧义时才问用户 |
| 审查线程直接修复问题 | 要求审查线程停止修改，只提交建议；由主线程重新分派修复 |
